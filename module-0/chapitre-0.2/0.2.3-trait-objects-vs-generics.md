# 0.2.3 — Trait Objects vs Generics : Analyse de performance et choix stratégiques

> **Prérequis :** [0.2.2 — Traits & Generics](./0.2.2-traits-generics.md)
> **Durée estimée :** 50 minutes

---

## Le coût de l'abstraction en Rust

Rust promet des "zero-cost abstractions" — l'idée que les abstractions haut niveau ne coûtent rien par rapport au code bas niveau équivalent. C'est vrai pour les generics. Ce n'est pas tout à fait vrai pour les trait objects. Comprendre la différence vous permet de choisir la bonne abstraction pour chaque situation.

---

## Monomorphisation — Comment les generics deviennent du code concret

Quand vous écrivez une fonction générique, le compilateur génère une **version spécialisée** pour chaque type concret utilisé :

```rust
fn afficher<T: std::fmt::Display>(val: T) {
    println!("{}", val);
}

fn main() {
    afficher(42);        // Le compilateur génère afficher_i32()
    afficher(3.14);      // Le compilateur génère afficher_f64()
    afficher("bonjour"); // Le compilateur génère afficher_str()
}
```

Le code généré est **identique** à si vous aviez écrit trois fonctions séparées. Zéro overhead à l'exécution.

**Avantage :** Le compilateur peut inliner les appels, optimiser pour chaque type.
**Inconvénient :** Temps de compilation plus long, binaire plus grand (code dupliqué).

---

## Vtables — Comment les trait objects fonctionnent

Les trait objects (`dyn Trait`) utilisent une **vtable** (virtual function table) — une table de pointeurs de fonctions associée à chaque type concret.

```
Structure d'un &dyn Compression :
┌──────────────┬──────────────────────────────┐
│ ptr données  │ ptr vtable                   │
└──────────────┴──────────────────────────────┘
                        │
                        ▼
                ┌──────────────────┐
                │ type_id          │
                │ taille           │
                │ drop()    ──────>│ fn drop_gzip()
                │ compresser() ───>│ fn compresser_gzip()
                └──────────────────┘
```

Chaque appel via `dyn Trait` :
1. Charge le pointeur de vtable
2. Charge l'adresse de la fonction dans la vtable
3. Saute à cette adresse

C'est **deux indirections** supplémentaires. Le compilateur ne peut pas inliner ces appels (le type est inconnu à la compilation).

## Benchmark : Generics vs Trait Objects

```rust
// Pour benchmarker, ajouter criterion dans Cargo.toml :
// [dev-dependencies]
// criterion = { version = "0.5", features = ["html_reports"] }

use criterion::{black_box, criterion_group, criterion_main, Criterion};

trait Calcul {
    fn calculer(&self, x: f64) -> f64;
}

struct Carré;
struct Racine;

impl Calcul for Carré   { fn calculer(&self, x: f64) -> f64 { x * x } }
impl Calcul for Racine  { fn calculer(&self, x: f64) -> f64 { x.sqrt() } }

// Version générique — dispatch statique
fn somme_générique<C: Calcul>(calc: &C, données: &[f64]) -> f64 {
    données.iter().map(|&x| calc.calculer(x)).sum()
}

// Version dynamique — dispatch via vtable
fn somme_dynamique(calc: &dyn Calcul, données: &[f64]) -> f64 {
    données.iter().map(|&x| calc.calculer(x)).sum()
}

fn benchmark(c: &mut Criterion) {
    let données: Vec<f64> = (0..10_000).map(|i| i as f64).collect();
    let calc = Carré;
    
    c.bench_function("générique", |b| {
        b.iter(|| somme_générique(&calc, black_box(&données)))
    });
    
    c.bench_function("dynamique", |b| {
        b.iter(|| somme_dynamique(&calc, black_box(&données)))
    });
}

criterion_group!(benches, benchmark);
criterion_main!(benches);
```

Résultats typiques :
```
générique   time: [12.3 µs 12.4 µs 12.5 µs]
dynamique   time: [23.1 µs 23.4 µs 23.7 µs]
```

Le dispatch dynamique est environ **2x plus lent** sur les appels fréquents. Pour des opérations coûteuses (I/O, réseau), cette différence est négligeable. Pour des calculs CPU intensifs en boucle serrée, elle compte.

## Guide de décision

```
┌──────────────────────────────────────────────────────┐
│ Le type est-il connu à la compilation ?              │
│ ├─ Oui ──> Generics (si performance critique)        │
│ └─ Non ──> dyn Trait obligatoire                     │
│                                                      │
│ Liste hétérogène de types différents ?               │
│ └─ Oui ──> Vec<Box<dyn Trait>>                       │
│                                                      │
│ Plugin chargé dynamiquement (WASM, FFI) ?            │
│ └─ Oui ──> dyn Trait                                 │
│                                                      │
│ Chemin chaud CPU avec millions d'appels/sec ?        │
│ └─ Oui ──> Generics absolument                       │
│                                                      │
│ API publique d'une bibliothèque ?                    │
│ └─ Privilégiez Generics (plus flexible pour l'appelant) │
└──────────────────────────────────────────────────────┘
```

## Pattern hybride : enum dispatch

Pour les performances critiques avec un nombre fini de variants connus :

```rust
// Souvent plus rapide que dyn Trait, plus flexible que pure generic
enum Sérialiseur {
    Json(JsonSérialiseur),
    MessagePack(MsgPackSérialiseur),
    Protobuf(ProtobufSérialiseur),
}

impl Sérialiseur {
    fn sérialiser(&self, données: &impl serde::Serialize) -> Vec<u8> {
        match self {
            Sérialiseur::Json(s) => s.sérialiser(données),
            Sérialiseur::MessagePack(s) => s.sérialiser(données),
            Sérialiseur::Protobuf(s) => s.sérialiser(données),
        }
    }
}
```

Le dispatch par `match` est compilé en branchement direct — souvent aussi rapide que le générique, plus flexible que le trait object.

---

## `impl Trait` en position de retour

Une syntaxe moderne qui cache le type concret tout en gardant le dispatch statique :

```rust
// Retourne "quelque chose qui itère sur des u32"
// Le type exact est caché — mais statique (pas de vtable)
fn générer_pairs(limite: u32) -> impl Iterator<Item = u32> {
    (0..limite).filter(|n| n % 2 == 0)
}

// impl Trait en paramètre — sucre syntaxique pour le generic bound
fn traiter(données: impl AsRef<[u8]>) -> usize {
    données.as_ref().len()
}
```

## Résumé

| Approche | Performance | Flexibilité | Taille binaire |
|---|---|---|---|
| Generics | ⚡ Maximale | 🔒 Type fixe à la compile | 📦 Plus grand |
| `dyn Trait` | 🐢 Overhead vtable | ✅ Type dynamique | 📦 Plus petit |
| Enum dispatch | ⚡ Très bonne | 🔒 Variants fixes | 📦 Moyen |
| `impl Trait` | ⚡ Maximale | 🔒 Type opaque mais statique | 📦 Plus grand |
