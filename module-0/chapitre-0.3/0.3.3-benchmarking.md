## La règle numéro un de l'optimisation

> *"Premature optimization is the root of all evil."* — Donald Knuth

Avant d'optimiser quoi que ce soit, **mesurez**. Un code qui semble lent peut être parfaitement acceptable. Un code qui semble correct peut être le vrai goulot d'étranglement. Les benchmarks transforment les intuitions en données.

## Criterion — Benchmarks fiables en Rust

`criterion` est le standard pour les benchmarks Rust. Il gère les warm-up, les statistiques et la détection de régressions.

```toml
# Cargo.toml
[dev-dependencies]
criterion = { version = "0.5", features = ["html_reports"] }

[[bench]]
name = "mes_benchmarks"
harness = false
```

```rust
// benches/mes_benchmarks.rs
use criterion::{black_box, criterion_group, criterion_main, Criterion, BenchmarkId};

// Fonction à benchmarker
fn recherche_linéaire(données: &[i32], cible: i32) -> Option<usize> {
    données.iter().position(|&x| x == cible)
}

fn recherche_binaire(données: &[i32], cible: i32) -> Option<usize> {
    données.binary_search(&cible).ok()
}

fn benchmark_recherche(c: &mut Criterion) {
    let mut groupe = c.benchmark_group("recherche");
    
    // Tester plusieurs tailles de données
    for taille in [100usize, 1_000, 10_000, 100_000] {
        let données: Vec<i32> = (0..taille as i32).collect();
        let cible = taille as i32 / 2;
        
        groupe.bench_with_input(
            BenchmarkId::new("linéaire", taille),
            &taille,
            |b, _| b.iter(|| recherche_linéaire(black_box(&données), black_box(cible)))
        );
        
        groupe.bench_with_input(
            BenchmarkId::new("binaire", taille),
            &taille,
            |b, _| b.iter(|| recherche_binaire(black_box(&données), black_box(cible)))
        );
    }
    
    groupe.finish();
}

criterion_group!(benches, benchmark_recherche);
criterion_main!(benches);
```

```bash
# Lancer les benchmarks
cargo bench

# Avec rapport HTML
cargo bench -- --output-format html
# Ouvrir target/criterion/report/index.html
```

Résultats typiques :
```
recherche/linéaire/100       time: [312 ns  314 ns  316 ns]
recherche/linéaire/100000    time: [1.2 ms  1.2 ms  1.3 ms]
recherche/binaire/100        time: [15 ns   15 ns   16 ns]
recherche/binaire/100000     time: [23 ns   24 ns   24 ns]
```

> 💡 `black_box()` empêche le compilateur d'optimiser le benchmark en éliminant le code "inutilisé". Sans lui, le compilateur pourrait calculer le résultat à la compilation et le benchmark mesurerait zéro.

## Détecter les régressions de performance

```bash
# Sauvegarder une baseline
cargo bench -- --save-baseline avant-optimisation

# Après vos modifications :
cargo bench -- --baseline avant-optimisation
```

Criterion comparera automatiquement et signalera toute régression.

## Profiling avec perf (Linux)

Pour les problèmes de performance en production, `perf` et les flamegraphs sont indispensables.

```bash
# Installer les outils
sudo apt install linux-perf
cargo install flamegraph

# Compiler avec symboles de debug (sans perdre les optimisations)
# Dans Cargo.toml :
# [profile.release]
# debug = true

# Générer un flamegraph
cargo flamegraph --bin mon-programme

# Ou avec perf directement
perf record --call-graph dwarf ./target/release/mon-programme
perf report
```

### Lire un flamegraph

```
         main (5%)
    ┌────┴────────────────────────────────┐
    traiter_requêtes (95%)
    ┌────────────┬──────────────────────┐
    parser (30%) sérialiser (65%)
                 ┌───────────────────┐
                 json::to_string (65%)
                 ┌──────────────────┐
                 alloc (40%)
```

La largeur = le temps CPU. Un pic large et profond = goulot d'étranglement identifié.

## Profiling mémoire avec heaptrack

```bash
# Installation
sudo apt install heaptrack heaptrack-gui

# Analyser
heaptrack ./target/debug/mon-programme
heaptrack_gui heaptrack.mon-programme.gz
```

Heaptrack montre :
- Les allocations heap par fonction
- Les allocations temporaires (créées et libérées rapidement)
- Les pics de consommation mémoire

## Benchmarks inline rapides avec `std::time`

Pour des mesures rapides sans setup :

```rust
use std::time::Instant;

fn main() {
    let données: Vec<i32> = (0..1_000_000).collect();
    
    let début = Instant::now();
    let somme: i64 = données.iter().map(|&x| x as i64).sum();
    let durée = début.elapsed();
    
    println!("Somme: {} en {:?}", somme, durée);
    // Somme: 499999500000 en 1.2ms
}
```

> ⚠️ Cette approche mesure une seule exécution — les résultats varient selon la charge système. Pour des mesures fiables, utilisez toujours criterion.

## 🔥 Failure First — Les optimisations prématurées

Voici un exemple classique d'optimisation mal ciblée :

```rust
// Avant "optimisation" — code clair, O(n)
fn compter_pairs(données: &[i32]) -> usize {
    données.iter().filter(|&&x| x % 2 == 0).count()
}

// Après "optimisation" — code complexe, même complexité
fn compter_pairs_optimisé(données: &[i32]) -> usize {
    let mut compte = 0usize;
    let mut i = 0;
    while i < données.len() {
        if données[i] & 1 == 0 { compte += 1; }
        i += 1;
    }
    compte
}
```

Benchmark réel sur 1M éléments :
```
compter_pairs          time: [890 µs 892 µs 895 µs]
compter_pairs_optimisé time: [889 µs 891 µs 893 µs]
```

Résultat : **zéro différence**. Le compilateur Rust optimise les deux versions identiquement. Vous avez complexifié le code pour rien.

**Leçon :** Mesurez d'abord. Optimisez ensuite. Et préférez la lisibilité jusqu'à ce que les données prouvent qu'il faut faire autrement.

## 🛠️ Profil de compilation release

```toml
# Cargo.toml — profil de production optimisé
[profile.release]
opt-level = 3         # Optimisation maximale
lto = "thin"          # Link-time optimization
codegen-units = 1     # Un seul codegen unit — plus lent à compiler, plus rapide à l'exécution
panic = "abort"       # Pas de stack unwinding — binaire plus petit
strip = true          # Supprimer les symboles de debug

# Profil pour les benchmarks — optimisé mais avec symboles
[profile.bench]
debug = true
```
