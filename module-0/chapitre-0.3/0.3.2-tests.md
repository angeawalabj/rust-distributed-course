## Pourquoi tester en Rust plus qu'ailleurs ?

Dans les langages avec garbage collector et typage dynamique, les tests servent souvent à détecter des bugs que le langage aurait pu prévenir (NullPointerException, type errors, memory corruption). En Rust, le compilateur élimine déjà cette catégorie de bugs.

Les tests en Rust servent donc à vérifier ce que le compilateur ne peut pas — la **logique métier** : est-ce que votre algorithme produit les bons résultats ? Est-ce que vos cas limites sont gérés correctement ? Est-ce que votre API se comporte comme documenté ?

## Tests unitaires — Dans le même fichier

```rust
// src/calculs.rs

pub fn diviser(a: f64, b: f64) -> Result<f64, &'static str> {
    if b == 0.0 {
        return Err("Division par zéro");
    }
    Ok(a / b)
}

pub fn est_premier(n: u64) -> bool {
    if n < 2 { return false; }
    if n == 2 { return true; }
    if n % 2 == 0 { return false; }
    
    let limite = (n as f64).sqrt() as u64;
    (3..=limite).step_by(2).all(|i| n % i != 0)
}

// Module de test — compilé seulement avec `cargo test`
#[cfg(test)]
mod tests {
    use super::*; // Importe tout du module parent
    
    // Tests de diviser()
    #[test]
    fn diviser_valeurs_normales() {
        assert_eq!(diviser(10.0, 2.0), Ok(5.0));
        assert_eq!(diviser(7.0, 2.0), Ok(3.5));
    }
    
    #[test]
    fn diviser_par_zéro_retourne_erreur() {
        assert!(diviser(5.0, 0.0).is_err());
        assert_eq!(diviser(5.0, 0.0), Err("Division par zéro"));
    }
    
    // Tests de est_premier()
    #[test]
    fn nombres_premiers_corrects() {
        let premiers = [2, 3, 5, 7, 11, 13, 17, 19, 23];
        for &n in &premiers {
            assert!(est_premier(n), "{} devrait être premier", n);
        }
    }
    
    #[test]
    fn nombres_composés_corrects() {
        let composés = [4, 6, 8, 9, 10, 12, 15, 100];
        for &n in &composés {
            assert!(!est_premier(n), "{} ne devrait pas être premier", n);
        }
    }
    
    #[test]
    fn cas_limites_premiers() {
        assert!(!est_premier(0));
        assert!(!est_premier(1));
        assert!(est_premier(2)); // Le seul pair premier
    }
}
```

## Macros d'assertion disponibles

```rust
#[test]
fn démonstration_assertions() {
    // Égalité
    assert_eq!(2 + 2, 4);
    assert_ne!(2 + 2, 5);
    
    // Condition
    assert!(true);
    assert!(!false);
    
    // Avec message d'erreur personnalisé
    let résultat = calculer(42);
    assert_eq!(
        résultat, 84,
        "calculer(42) devrait retourner 84, a retourné {:?}",
        résultat
    );
    
    // Flottants — éviter assert_eq! (précision)
    let résultat: f64 = 0.1 + 0.2;
    assert!((résultat - 0.3).abs() < 1e-10, "0.1 + 0.2 devrait être proche de 0.3");
    // Ou avec approx:
    // assert_abs_diff_eq!(résultat, 0.3, epsilon = 1e-10);
}
```

## Tests d'intégration — Dossier tests/

Les tests d'intégration testent votre bibliothèque comme un utilisateur externe :

```
mon-projet/
├── src/
│   └── lib.rs
├── tests/
│   ├── intégration_basique.rs
│   ├── intégration_erreurs.rs
│   └── commun/
│       └── mod.rs   ← helpers partagés entre les fichiers de test
```

```rust
// tests/commun/mod.rs
pub fn créer_store_de_test() -> gestion_utilisateurs::StoreUtilisateurs {
    let mut store = gestion_utilisateurs::StoreUtilisateurs::nouveau();
    store.créer("Admin Test", "admin@test.com", gestion_utilisateurs::Rôle::Administrateur).unwrap();
    store
}
```

```rust
// tests/intégration_basique.rs
mod commun;

use gestion_utilisateurs::Rôle;

#[test]
fn flux_complet_utilisateur() {
    let mut store = commun::créer_store_de_test();
    
    // 1. Créer
    let user = store.créer("Alice", "alice@test.com", Rôle::Utilisateur).unwrap();
    let id = user.id;
    
    // 2. Lire
    let trouvé = store.obtenir_par_id(id).unwrap();
    assert_eq!(trouvé.nom, "Alice");
    
    // 3. Mettre à jour
    let mis_à_jour = store.mettre_à_jour_nom(id, "Alice Dupont").unwrap();
    assert_eq!(mis_à_jour.nom, "Alice Dupont");
    
    // 4. Supprimer
    store.supprimer(id).unwrap();
    
    // 5. Vérifier l'état supprimé
    let supprimé = store.obtenir_par_id(id).unwrap();
    assert!(!supprimé.est_actif());
}
```

## Mocking avec des traits

Rust n'a pas de mocking magique comme Mockito (Java) ou unittest.mock (Python). La bonne approche : utiliser des traits pour l'injection de dépendances.

```rust
// Trait pour la couche de données — permet de mock en test
trait StockageTrait {
    fn obtenir(&self, id: u64) -> Option<String>;
    fn sauvegarder(&mut self, id: u64, valeur: String) -> bool;
}

// Implémentation réelle
struct StockageBDD { /* connexion DB */ }
impl StockageTrait for StockageBDD { /* ... */ }

// Mock pour les tests
struct StockageMock {
    données: std::collections::HashMap<u64, String>,
    écritures: Vec<(u64, String)>, // Trace les appels
}

impl StockageTrait for StockageMock {
    fn obtenir(&self, id: u64) -> Option<String> {
        self.données.get(&id).cloned()
    }
    
    fn sauvegarder(&mut self, id: u64, valeur: String) -> bool {
        self.écritures.push((id, valeur.clone()));
        self.données.insert(id, valeur);
        true
    }
}

// Service générique sur le trait
struct ServiceUtilisateur<S: StockageTrait> {
    stockage: S,
}

impl<S: StockageTrait> ServiceUtilisateur<S> {
    fn obtenir_ou_défaut(&self, id: u64) -> String {
        self.stockage.obtenir(id).unwrap_or_else(|| "inconnu".to_string())
    }
}

#[cfg(test)]
mod tests {
    use super::*;
    
    #[test]
    fn obtenir_ou_défaut_avec_mock() {
        let mut stockage = StockageMock {
            données: std::collections::HashMap::new(),
            écritures: vec![],
        };
        stockage.données.insert(1, "Alice".to_string());
        
        let service = ServiceUtilisateur { stockage };
        
        assert_eq!(service.obtenir_ou_défaut(1), "Alice");
        assert_eq!(service.obtenir_ou_défaut(99), "inconnu");
    }
}
```

## Lancer les tests

```bash
# Tous les tests
cargo test

# Un test spécifique
cargo test test_flux_complet

# Tests d'un module spécifique
cargo test calculs::tests

# Avec output (println! visible)
cargo test -- --nocapture

# Tests ignorés
cargo test -- --ignored

# Parallélisme contrôlé (utile si tests avec ressources partagées)
cargo test -- --test-threads=1

# Tests d'intégration seulement
cargo test --test intégration_basique
```

## Résumé

| Type de test | Emplacement | Quand l'utiliser |
|---|---|---|
| Unitaire | `src/` dans `#[cfg(test)]` | Fonctions isolées, logique pure |
| Intégration | `tests/` | Flux complets, API publique |
| Documentation | Dans les doc comments | Exemples dans la doc |
| Benchmark | `benches/` avec criterion | Mesure de performance |
