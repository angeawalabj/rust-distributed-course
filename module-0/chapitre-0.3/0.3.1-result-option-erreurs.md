## Pourquoi la gestion d'erreurs est cruciale en systèmes distribués

Dans un système distribué, les erreurs ne sont pas l'exception — elles sont la norme. Le réseau tombe, la base de données refuse une connexion, un fichier de configuration est mal formé, un service tiers répond avec un statut 503. Un système bien conçu **anticipe** ces situations et les gère gracieusement.

Rust vous force à affronter les erreurs là où elles se produisent. Il n'y a pas d'exceptions silencieuses, pas de valeurs null inattendues — chaque chemin d'erreur doit être traité explicitement.

C'est inconfortable au début. C'est précieux en production.

## Option\<T\> — L'absence explicite

`Option<T>` représente une valeur qui peut exister ou non. C'est le remplacement de `null` — avec la différence que le compilateur vous force à vérifier avant d'utiliser la valeur.

```rust
enum Option<T> {
    Some(T), // La valeur existe
    None,    // Pas de valeur
}
```

### Utilisation de base

```rust
fn trouver_utilisateur(id: u64) -> Option<String> {
    let base = vec![
        (1u64, "Alice"),
        (2u64, "Bob"),
        (3u64, "Charlie"),
    ];
    
    base.iter()
        .find(|(uid, _)| *uid == id)
        .map(|(_, nom)| nom.to_string())
}

fn main() {
    // Traitement explicite
    match trouver_utilisateur(1) {
        Some(nom) => println!("Trouvé : {}", nom),
        None => println!("Utilisateur introuvable"),
    }
    
    // Avec if let — plus concis quand on ignore None
    if let Some(nom) = trouver_utilisateur(2) {
        println!("Utilisateur 2 : {}", nom);
    }
    
    // Combinateurs fonctionnels — chaînage élégant
    let longueur = trouver_utilisateur(3)
        .map(|nom| nom.len())
        .unwrap_or(0);
    
    println!("Longueur du nom : {}", longueur);
}
```

### Méthodes utiles d'Option

```rust
let opt: Option<i32> = Some(42);

opt.unwrap();              // 42, ou panic si None
opt.unwrap_or(0);          // 42, ou 0 si None
opt.unwrap_or_default();   // 42, ou valeur par défaut du type
opt.unwrap_or_else(|| calculer_défaut()); // Lazy — calculé seulement si None

opt.map(|v| v * 2);        // Some(84), ou None si opt était None
opt.and_then(|v| if v > 0 { Some(v) } else { None }); // "flatmap"
opt.filter(|v| *v > 10);   // Some(42), ou None si condition fausse
opt.is_some();             // true
opt.is_none();             // false
opt.ok_or("manquant");     // Convertit en Result<i32, &str>
```

## Result\<T, E\> — Les erreurs récupérables

`Result<T, E>` représente soit un succès (`Ok(T)`) soit une erreur (`Err(E)`).

```rust
enum Result<T, E> {
    Ok(T),   // Succès avec valeur de type T
    Err(E),  // Erreur de type E
}
```

### Utilisation de base

```rust
use std::num::ParseIntError;

fn parser_port(s: &str) -> Result<u16, ParseIntError> {
    s.trim().parse::<u16>()
}

fn main() {
    match parser_port("8080") {
        Ok(port) => println!("Port valide : {}", port),
        Err(e) => eprintln!("Port invalide : {}", e),
    }
    
    match parser_port("invalid") {
        Ok(port) => println!("Port : {}", port),
        Err(e) => eprintln!("Erreur : {}", e),
    }
}
```

## L'opérateur ? — Propagation d'erreurs

L'opérateur `?` est le pattern le plus important pour la gestion d'erreurs en Rust. Il **propage** automatiquement les erreurs vers l'appelant.

```rust
// Sans ? — verbeux et répétitif
fn lire_config_verbeux(chemin: &str) -> Result<Config, Box<dyn std::error::Error>> {
    let contenu = match std::fs::read_to_string(chemin) {
        Ok(c) => c,
        Err(e) => return Err(Box::new(e)),
    };
    
    let config = match serde_json::from_str(&contenu) {
        Ok(c) => c,
        Err(e) => return Err(Box::new(e)),
    };
    
    Ok(config)
}

// Avec ? — élégant et idiomatique
fn lire_config(chemin: &str) -> Result<Config, Box<dyn std::error::Error>> {
    let contenu = std::fs::read_to_string(chemin)?; // Propage l'erreur si Err
    let config: Config = serde_json::from_str(&contenu)?;
    Ok(config)
}
```

`?` fait exactement ceci :
1. Si `Ok(v)` → continue avec `v`
2. Si `Err(e)` → retourne `Err(e.into())` immédiatement (avec conversion automatique si `From` est implémenté)

## Erreurs personnalisées avec `thiserror`

Pour un code de production, créez vos propres types d'erreur :

```bash
# Dans Cargo.toml
[dependencies]
thiserror = "1"
```

```rust
use thiserror::Error;

#[derive(Error, Debug)]
enum ErreurApplication {
    #[error("Erreur de lecture du fichier {chemin}: {source}")]
    FichierInaccessible {
        chemin: String,
        #[source]
        source: std::io::Error,
    },
    
    #[error("Configuration invalide : {0}")]
    ConfigInvalide(String),
    
    #[error("Base de données indisponible")]
    BaseDeDonnées(#[from] sqlx::Error),
    
    #[error("Erreur réseau sur {url}")]
    Réseau {
        url: String,
        #[source]
        source: reqwest::Error,
    },
}

// Maintenant vous pouvez propager avec ?
async fn initialiser() -> Result<(), ErreurApplication> {
    let config_texte = std::fs::read_to_string("config.json")
        .map_err(|e| ErreurApplication::FichierInaccessible {
            chemin: "config.json".to_string(),
            source: e,
        })?;
    
    // ...
    Ok(())
}
```

## `anyhow` — Erreurs sans effort pour les binaires

Pour les applications (vs les bibliothèques), `anyhow` simplifie la gestion d'erreurs :

```bash
[dependencies]
anyhow = "1"
```

```rust
use anyhow::{Context, Result, bail, ensure};

fn traiter_requete(id: &str) -> Result<String> {
    // Context ajoute du contexte à n'importe quelle erreur
    let n: u64 = id.parse()
        .context("L'identifiant doit être un nombre entier")?;
    
    // bail! retourne une erreur immédiatement
    if n == 0 {
        bail!("L'identifiant 0 est réservé");
    }
    
    // ensure! vérifie une condition
    ensure!(n < 1_000_000, "Identifiant trop grand : {}", n);
    
    let résultat = chercher_en_base(n)
        .with_context(|| format!("Impossible de trouver l'entrée {}", n))?;
    
    Ok(résultat)
}

// Dans main(), affichage automatique de la chaîne d'erreurs
fn main() -> Result<()> {
    let résultat = traiter_requete("abc")?;
    println!("{}", résultat);
    Ok(())
}
```

## Bibliothèques vs Binaires — La règle d'or

> **Bibliothèques :** utilisez `thiserror` et des types d'erreur concrets.
> **Binaires/Applications :** utilisez `anyhow` pour la flexibilité.

```rust
// lib.rs — bibliothèque : erreurs précises pour les appelants
pub fn calculer(input: &str) -> Result<f64, ErreurCalcul> { ... }

// main.rs — application : juste afficher les erreurs clairement
fn main() -> anyhow::Result<()> {
    let result = ma_lib::calculer("input")?;
    Ok(())
}
```

## 🔥 Failure First — Le piège du `.unwrap()` en production

```rust
// ❌ Code qui va exploser en production
fn traiter_paiement(montant_str: &str) -> f64 {
    let montant: f64 = montant_str.parse().unwrap(); // Panic si l'entrée est invalide
    let taux = std::env::var("TAUX_TVA").unwrap();   // Panic si la variable manque
    let taux_f: f64 = taux.parse().unwrap();          // Panic si la valeur est invalide
    montant * (1.0 + taux_f)
}
```

En production avec un input utilisateur malveillant ou une variable d'environnement manquante, ce code fait **crasher votre service**.

```rust
// ✅ Version production-ready
use thiserror::Error;

#[derive(Error, Debug)]
enum ErreurPaiement {
    #[error("Montant invalide '{0}' : doit être un nombre")]
    MontantInvalide(String),
    #[error("Variable d'environnement TAUX_TVA manquante")]
    TauxManquant,
    #[error("Taux TVA invalide '{0}' : doit être un nombre")]
    TauxInvalide(String),
}

fn traiter_paiement(montant_str: &str) -> Result<f64, ErreurPaiement> {
    let montant: f64 = montant_str.parse()
        .map_err(|_| ErreurPaiement::MontantInvalide(montant_str.to_string()))?;
    
    let taux_str = std::env::var("TAUX_TVA")
        .map_err(|_| ErreurPaiement::TauxManquant)?;
    
    let taux: f64 = taux_str.parse()
        .map_err(|_| ErreurPaiement::TauxInvalide(taux_str.clone()))?;
    
    Ok(montant * (1.0 + taux))
}
```

## Structurer les erreurs dans une application multi-couches

```rust
// Couche données
#[derive(Error, Debug)]
pub enum ErreurDonnées {
    #[error("Enregistrement introuvable : id={0}")]
    Introuvable(u64),
    #[error("Connexion base de données : {0}")]
    Connexion(#[from] sqlx::Error),
}

// Couche service
#[derive(Error, Debug)]
pub enum ErreurService {
    #[error("Données : {0}")]
    Données(#[from] ErreurDonnées),
    #[error("Validation échouée : {0}")]
    Validation(String),
    #[error("Limite de taux dépassée")]
    RateLimitDépassé,
}

// Couche API — erreur finale avec code HTTP
#[derive(Error, Debug)]
pub enum ErreurAPI {
    #[error("Service : {0}")]
    Service(#[from] ErreurService),
    #[error("Authentification requise")]
    NonAuthentifié,
    #[error("Interdit")]
    Interdit,
}

impl ErreurAPI {
    pub fn code_http(&self) -> u16 {
        match self {
            ErreurAPI::NonAuthentifié => 401,
            ErreurAPI::Interdit => 403,
            ErreurAPI::Service(ErreurService::Validation(_)) => 400,
            ErreurAPI::Service(ErreurService::RateLimitDépassé) => 429,
            ErreurAPI::Service(ErreurService::Données(ErreurDonnées::Introuvable(_))) => 404,
            _ => 500,
        }
    }
}
```

## Résumé

| Outil | Usage | Package |
|---|---|---|
| `Option<T>` | Valeur absente possible | `std` |
| `Result<T, E>` | Opération qui peut échouer | `std` |
| `?` | Propagation élégante | `std` |
| `thiserror` | Erreurs dans les bibliothèques | `thiserror` |
| `anyhow` | Erreurs dans les applications | `anyhow` |
| `.context()` | Ajouter du contexte | `anyhow` |

**Règle absolue de production :** Zéro `.unwrap()` sur des données externes. Toujours `.context()` ou `map_err()` pour donner du sens aux erreurs.
