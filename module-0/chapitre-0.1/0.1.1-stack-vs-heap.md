# 0.1.1 — Stack vs Heap : Comprendre la mémoire pour optimiser ses programmes

> **Prérequis :** Aucun — c'est le point de départ.
> **Durée estimée :** 45 minutes

---

## Pourquoi comprendre la mémoire change tout

En Python ou JavaScript, vous créez des objets et le runtime s'occupe de tout. La mémoire s'alloue, se désalloue, et vous n'y pensez presque jamais. C'est pratique — mais cette abstraction a un coût : des pauses GC imprévisibles, une consommation mémoire difficile à contrôler, et une incapacité à raisonner précisément sur les performances.

En Rust, vous n'avez pas de garbage collector. Vous devez comprendre où vit chaque valeur en mémoire, combien de temps elle vit, et qui en est responsable. En échange, vous obtenez des performances prévisibles, une empreinte mémoire minimale, et un compilateur qui garantit l'absence d'erreurs mémoire.

Commençons par les deux régions de mémoire que tout programme utilise : la **stack** et le **heap**.

---

## La Stack — rapide et ordonnée

La stack (pile) est une zone mémoire organisée comme son nom l'indique : en pile. Les valeurs s'empilent et se désempilent dans un ordre strict — LIFO (Last In, First Out).

### Caractéristiques de la stack

- **Allocation instantanée :** déplacer un pointeur de pile de N octets, c'est tout ce qu'il faut faire.
- **Taille connue à la compilation :** seules les valeurs dont on connaît la taille statiquement peuvent vivre sur la stack.
- **Durée de vie automatique :** une valeur sur la stack disparaît dès que la fonction qui l'a créée se termine.
- **Taille limitée :** généralement 8 Mo par thread sur Linux. Dépasser cette limite provoque un stack overflow.

```rust
fn main() {
    // Ces valeurs vivent sur la stack
    let x: i32 = 42;          // 4 octets sur la stack
    let y: f64 = 3.14;        // 8 octets sur la stack
    let flag: bool = true;    // 1 octet sur la stack
    
    // Un tableau de taille fixe : aussi sur la stack
    let arr: [i32; 5] = [1, 2, 3, 4, 5]; // 20 octets sur la stack
    
    println!("x={}, y={}, flag={}", x, y, flag);
} // ← Toutes ces valeurs sont automatiquement libérées ici
```

Quand `main()` se termine, le pointeur de pile revient à sa position initiale. Aucun appel à `free()`, aucun garbage collector — c'est instantané et gratuit.

### Visualisation de la stack

```
Appel de fonction : calculer(a: i32, b: i32)
┌─────────────────────────────┐  ← sommet de la pile (Stack Pointer)
│  résultat : i32  (4 octets) │
│  b : i32         (4 octets) │
│  a : i32         (4 octets) │
├─────────────────────────────┤  ← frame de la fonction appelante
│  ...variables de main()...  │
└─────────────────────────────┘  ← base de la pile
```

Chaque appel de fonction crée un nouveau "frame" sur la pile. Quand la fonction retourne, son frame est dépilé.

## Le Heap — flexible mais coûteux

Le heap (tas) est une zone mémoire dynamique. Il n'a pas d'ordre particulier — les allocations peuvent se faire et se défaire dans n'importe quel ordre. C'est là que vivent les données dont la taille n'est pas connue à la compilation, ou qui doivent vivre plus longtemps que la fonction qui les crée.

### Caractéristiques du heap

- **Allocation dynamique :** l'allocateur (jemalloc, mimalloc, etc.) cherche un bloc libre de la bonne taille.
- **Taille variable :** un `Vec<T>` peut grandir ou rétrécir à l'exécution.
- **Durée de vie explicite :** en Rust, l'ownership détermine quand la mémoire est libérée.
- **Plus lent :** une allocation heap implique un appel système et peut fragmenter la mémoire.

```rust
fn main() {
    // String alloue sur le heap (taille variable)
    let mut s = String::from("bonjour"); // heap: "bonjour" + capacité + longueur
    s.push_str(", monde");               // peut réallouer si nécessaire
    
    // Vec alloue sur le heap
    let mut v: Vec<i32> = Vec::new();
    v.push(1);
    v.push(2);
    v.push(3);
    
    println!("{}", s);
    println!("{:?}", v);
} // ← Drop appelé automatiquement sur s et v → libération du heap
```

### La structure d'un String en mémoire

```
Stack                    Heap
┌──────────────┐        ┌─────────────────────┐
│ ptr ─────────┼───────>│ b o n j o u r , ...  │
│ len: 13      │        └─────────────────────┘
│ capacity: 16 │
└──────────────┘
```

La variable `s` elle-même (trois mots machine : pointeur, longueur, capacité) vit sur la **stack**. Les données "bonjour, monde" vivent sur le **heap**. C'est un pattern fondamental en Rust.

## Stack vs Heap — tableau comparatif

| Critère | Stack | Heap |
|---|---|---|
| Vitesse d'allocation | ⚡ Instantanée | 🐢 Coût variable |
| Taille connue | ✅ Obligatoire à la compile | ✅ Peut être dynamique |
| Durée de vie | Liée à la fonction | Contrôlée par ownership |
| Limite de taille | ~8 Mo (par thread) | Limitée par la RAM |
| Fragmentation | ❌ Impossible | ⚠️ Possible |
| Thread-safe | ✅ Par construction | ⚠️ Nécessite synchronisation |

## Quand utilise-t-on le heap en Rust ?

Rust vous laisse le choix — et il est important de le faire consciemment.

### Types qui allouent sur le heap

```rust
// String (vs &str qui est une référence vers un slice)
let owned: String = String::from("propriété");
let borrowed: &str = "référence statique";  // dans le segment de données, pas le heap

// Vec<T>
let numbers: Vec<i32> = vec![1, 2, 3, 4, 5];

// Box<T> — allocation heap explicite
let boxed: Box<i32> = Box::new(42);

// Les collections en général
use std::collections::HashMap;
let mut map: HashMap<String, i32> = HashMap::new();
```

### Types qui restent sur la stack

```rust
// Types primitifs
let n: i32 = 10;
let f: f64 = 3.14;
let b: bool = false;
let c: char = 'a';

// Tableaux de taille fixe
let arr: [u8; 256] = [0; 256];

// Structs dont tous les champs sont sur la stack
struct Point {
    x: f64,
    y: f64,
}
let p = Point { x: 1.0, y: 2.0 }; // 16 octets sur la stack
```

## 🔥 Failure First — Le stack overflow

Voyons ce qui se passe quand on abuse de la stack :

```rust
// ⚠️ Ce code va planter — c'est voulu !
fn recurse_infinitely(n: u64) -> u64 {
    // Chaque appel crée un nouveau frame sur la stack
    // Avec n=0, ça ne s'arrête jamais
    recurse_infinitely(n + 1)
}

fn main() {
    println!("Résultat : {}", recurse_infinitely(0));
}
```

Si vous compilez et exécutez ce code :
```
thread 'main' has overflowed its stack
fatal runtime error: stack overflow
```

**Diagnostic :** Rust détecte le dépassement et termine le thread proprement. Pas de corruption silencieuse.

**Correction :** Utiliser l'itération plutôt que la récursion, ou utiliser `Box` pour mettre les données sur le heap quand la profondeur de récursion est importante.

```rust
// Version correcte avec itération
fn sum_iterative(n: u64) -> u64 {
    (0..=n).sum()
}
```

## 🔬 Sous le capot — Le Drop trait

En Rust, quand une valeur quitte son scope, le compilateur insère automatiquement un appel à `drop()`. C'est le mécanisme qui remplace le GC :

```rust
struct ResourceHolder {
    name: String,
}

impl Drop for ResourceHolder {
    fn drop(&mut self) {
        // Appelé automatiquement à la fin du scope
        println!("Libération de : {}", self.name);
    }
}

fn main() {
    let r1 = ResourceHolder { name: String::from("connexion DB") };
    {
        let r2 = ResourceHolder { name: String::from("fichier log") };
        println!("Dans le bloc interne");
    } // ← r2.drop() appelé ici : "Libération de : fichier log"
    
    println!("Dans le bloc externe");
} // ← r1.drop() appelé ici : "Libération de : connexion DB"
```

Sortie :
```
Dans le bloc interne
Libération de : fichier log
Dans le bloc externe
Libération de : connexion DB
```

C'est le **RAII** (Resource Acquisition Is Initialization) — un pattern hérité de C++ et élevé au rang de principe fondamental en Rust. Toute ressource (mémoire, fichier, socket, mutex) est automatiquement libérée quand son propriétaire quitte le scope.

## 💡 Box<T> — Allocation heap explicite

Parfois, vous avez besoin de forcer une valeur sur le heap. `Box<T>` est le type pour ça :

```rust
fn main() {
    // Sans Box : l'arbre récursif ne peut pas être sur la stack
    // (taille inconnue à la compilation)
    enum Liste {
        Cons(i32, Box<Liste>),
        Nil,
    }
    
    use Liste::*;
    
    let liste = Cons(1, Box::new(Cons(2, Box::new(Cons(3, Box::new(Nil))))));
    
    // Box se déréférence transparemment
    let b = Box::new(5);
    println!("b = {}", *b);    // ou juste `b` dans la plupart des contextes
    println!("b + 1 = {}", *b + 1);
} // ← Box libère automatiquement la mémoire heap ici
```

**Quand utiliser `Box<T>` :**
- Types récursifs (arbres, listes chaînées)
- Données volumineuses à ne pas copier sur la stack
- Trait objects (`Box<dyn Trait>`)

## 🛠️ En production

Dans un serveur web haute performance, le choix stack/heap est critique :

```rust
// ❌ Moins efficace : alloue sur le heap à chaque requête
fn handle_request_slow(path: &str) -> String {
    let mut response = String::new();
    response.push_str("HTTP/1.1 200 OK\r\n");
    response.push_str("Content-Type: text/plain\r\n\r\n");
    response.push_str("Hello, World!");
    response
}

// ✅ Plus efficace : travaille avec des références et des slices
fn handle_request_fast(path: &str) -> &'static str {
    match path {
        "/health" => "OK",
        _ => "Not Found",
    }
}

// ✅ Pattern productif : pré-allouer avec capacité connue
fn build_response(body: &str) -> String {
    let estimated_size = 100 + body.len();
    let mut response = String::with_capacity(estimated_size);
    response.push_str("HTTP/1.1 200 OK\r\n\r\n");
    response.push_str(body);
    response
}
```

## Résumé

| À retenir | Pourquoi c'est important |
|---|---|
| La stack est rapide mais limitée | Utilisez-la pour les données de taille fixe |
| Le heap est flexible mais coûteux | Utilisez-le pour les données dynamiques |
| `Drop` remplace le GC | Les ressources sont libérées déterministiquement |
| `Box<T>` force l'allocation heap | Utile pour les types récursifs et les grandes données |
| Pas de GC = performances prévisibles | Critique pour les systèmes temps réel |
