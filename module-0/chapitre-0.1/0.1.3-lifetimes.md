# 0.1.3 — Lifetimes : Quand les références vivent-elles ?

> **Prérequis :** [0.1.2 — Ownership et Borrowing](./0.1.2-ownership-borrowing.md)
> **Durée estimée :** 60 minutes

## Le problème des dangling references

Une référence qui pointe vers une valeur détruite est appelée **dangling reference** (référence pendante). C'est l'une des causes les plus fréquentes de crashes et de vulnérabilités en C/C++. Rust les empêche complètement — grâce aux **lifetimes**.

Voici le scénario classique que Rust refuse :

```rust
fn reference_invalide() -> &String {
    let s = String::from("bonjour");
    &s // ← ERREUR : on retourne une référence vers s
       // mais s sera détruit à la fin de la fonction !
}
```

Le compilateur répond :
```
error[E0106]: missing lifetime specifier
  --> src/main.rs:1:26
   |
 1 | fn reference_invalide() -> &String {
   |                            ^ expected named lifetime parameter
```

---

## Qu'est-ce qu'une lifetime ?

Une **lifetime** est simplement la durée pendant laquelle une référence est valide. Chaque référence en Rust a une lifetime — mais dans la grande majorité des cas, le compilateur l'infère automatiquement et vous n'avez pas à la noter.

Les lifetimes deviennent visibles quand le compilateur ne peut pas les déduire seul, notamment quand des références entrent *et* sortent d'une fonction.

### Syntaxe des lifetimes

Les lifetimes s'écrivent avec une apostrophe suivie d'un nom : `'a`, `'b`, `'static`, etc.

```rust
// Syntaxe générale
fn ma_fonction<'a>(parametre: &'a str) -> &'a str {
    parametre
}
```

Ici, `'a` signifie : "la référence retournée vit au moins aussi longtemps que la référence passée en paramètre".

---

## L'inférence de lifetime

Le compilateur applique trois règles d'**élision** (inférence automatique) qui couvrent la majorité des cas :

**Règle 1 :** Chaque paramètre de référence reçoit sa propre lifetime.
```rust
fn f(x: &str, y: &str)  →  fn f<'a, 'b>(x: &'a str, y: &'b str)
```

**Règle 2 :** S'il y a exactement un paramètre référence, sa lifetime est assignée à la sortie.
```rust
fn f(x: &str) -> &str  →  fn f<'a>(x: &'a str) -> &'a str
```

**Règle 3 :** S'il y a un paramètre `&self` ou `&mut self`, sa lifetime est assignée à la sortie.

Quand ces règles ne suffisent pas, vous devez annoter manuellement.

## Lifetimes explicites — Quand et pourquoi

### Cas 1 : Fonction qui retourne l'une de ses références

```rust
// ❌ Ne compile pas — le compilateur ne sait pas quelle référence est retournée
fn plus_long(s1: &str, s2: &str) -> &str {
    if s1.len() > s2.len() { s1 } else { s2 }
}

// ✅ Avec annotation : "la sortie vit aussi longtemps que la plus courte des entrées"
fn plus_long<'a>(s1: &'a str, s2: &'a str) -> &'a str {
    if s1.len() > s2.len() { s1 } else { s2 }
}
```

L'annotation `'a` dit : "s1, s2, et le retour ont tous la même lifetime — le compilateur utilisera la plus courte".

```rust
fn main() {
    let string1 = String::from("longue chaîne");
    let resultat;
    {
        let string2 = String::from("xyz");
        resultat = plus_long(string1.as_str(), string2.as_str());
        println!("La plus longue : {}", resultat); // ✅ OK ici
    }
    // println!("{}", resultat); // ← ERREUR : string2 est détruite
}
```

### Cas 2 : Structs contenant des références

Si votre struct contient une référence, vous devez annoter la lifetime :

```rust
// La struct "Emprunseur" ne peut pas vivre plus longtemps que 'a
struct Emprunteur<'a> {
    extrait: &'a str,
}

impl<'a> Emprunteur<'a> {
    fn affiche(&self) {
        println!("Extrait : {}", self.extrait);
    }
}

fn main() {
    let texte = String::from("Voici un long texte de démonstration");
    
    let emprunteur = Emprunteur {
        extrait: &texte[0..5], // "Voici"
    };
    
    emprunteur.affiche(); // ✅
    
    // Si texte était droppé avant emprunteur, le compilateur refuserait
}
```

## 🔥 Failure First — Diagnostiquer une erreur de lifetime

```rust
fn titre_du_document<'a>(doc: &'a str, titre_par_defaut: &str) -> &'a str {
    let titre = titre_par_defaut.to_string(); // Nouvelle String sur le heap
    
    if doc.starts_with("# ") {
        &doc[2..]
    } else {
        &titre // ← ERREUR : titre ne vit que jusqu'à la fin de la fonction
               // mais on déclare que la sortie vit aussi longtemps que 'a
    }
}
```

Erreur :
```
error[E0515]: cannot return reference to local variable `titre`
```

**Correction :** Soit retourner une `String` (owned), soit changer la signature :

```rust
// Option 1 : retourner une String possédée
fn titre_du_document(doc: &str, titre_par_defaut: &str) -> String {
    if doc.starts_with("# ") {
        doc[2..].to_string()
    } else {
        titre_par_defaut.to_string()
    }
}

// Option 2 : faire en sorte que les deux références aient la même lifetime
fn titre_du_document<'a>(doc: &'a str, titre_par_defaut: &'a str) -> &'a str {
    if doc.starts_with("# ") {
        &doc[2..]
    } else {
        titre_par_defaut // On retourne la référence directement, pas une copie
    }
}
```

## La lifetime `'static`

`'static` est une lifetime spéciale : elle dure toute la durée d'exécution du programme.

```rust
// Les string literals sont 'static — elles vivent dans le binaire compilé
let s: &'static str = "Je vis pour toujours";

// Les constantes aussi
static CONSTANTE: &str = "Constante globale";

// On peut utiliser 'static comme contrainte de trait
fn doit_vivre_longtemps<T: 'static>(val: T) {
    // T ne peut pas contenir de références temporaires
    std::thread::spawn(move || {
        println!("{:?}", val); // val doit être 'static pour traverser un thread
    });
}
```

> ⚠️ **Piège courant :** ajouter `'static` partout pour faire taire le compilateur. C'est presque toujours la mauvaise solution — elle contraint artificiellement votre code.

## Lifetimes dans les méthodes

Avec `&self`, l'élision couvre la plupart des cas :

```rust
struct Analyseur {
    texte: String,
}

impl Analyseur {
    // Elision règle 3 : retour a la lifetime de &self
    fn premier_mot(&self) -> &str {
        match self.texte.find(' ') {
            Some(i) => &self.texte[..i],
            None => &self.texte[..],
        }
    }
    
    // Cas où l'annotation est nécessaire
    // "retourne la plus longue chaîne entre le texte interne et le paramètre"
    fn plus_long_avec<'a>(&'a self, autre: &'a str) -> &'a str {
        if self.texte.len() > autre.len() {
            &self.texte
        } else {
            autre
        }
    }
}
```

---

## 🔬 Sous le capot — Le borrow checker et les lifetimes

Internement, le compilateur représente chaque référence avec une région de code (un span) pendant laquelle elle est valide. Il vérifie ensuite que :

1. Aucune référence n'est utilisée après la destruction de sa cible.
2. Les lifetimes déclarées correspondent aux lifetimes réelles d'exécution.

Ce système est purement statique — il n'existe aucune information de lifetime à l'exécution. C'est ce qui le rend sans coût.

```
Analyse du compilateur pour plus_long<'a>(s1: &'a str, s2: &'a str) -> &'a str :

Appel : plus_long(string1.as_str(), string2.as_str())
  - Lifetime de string1 : jusqu'à la fin de main()
  - Lifetime de string2 : jusqu'à la fin du bloc interne
  - Lifetime de 'a : min(string1, string2) = jusqu'à la fin du bloc interne
  - Conclusion : le retour ne peut être utilisé que dans le bloc interne
```

## Patterns avancés

### Lifetime dans une struct avec méthode

```rust
struct Configuration<'a> {
    hote: &'a str,
    port: u16,
}

impl<'a> Configuration<'a> {
    fn nouvelle(hote: &'a str, port: u16) -> Self {
        Configuration { hote, port }
    }
    
    fn adresse(&self) -> String {
        format!("{}:{}", self.hote, self.port)
    }
    
    // Retourne une référence à l'hôte — lifetime liée à 'a (pas à &self)
    fn hote(&self) -> &'a str {
        self.hote
    }
}

fn main() {
    let adresse = String::from("localhost");
    let config = Configuration::nouvelle(&adresse, 8080);
    
    let hote = config.hote(); // hote vit aussi longtemps que adresse
    drop(config);             // config est droppé
    println!("{}", hote);     // ✅ encore valide — lié à adresse, pas à config
}
```

### Contraintes de lifetime multiples

```rust
// T doit implémenter Display ET vivre au moins aussi longtemps que 'a
fn affiche_ref<'a, T: std::fmt::Display + 'a>(val: &'a T) {
    println!("{}", val);
}
```

## Récapitulatif pratique

| Situation | Annotation nécessaire ? |
|---|---|
| Fonction avec un seul paramètre référence | ❌ Non (élision) |
| Méthode avec &self | ❌ Non (élision) |
| Fonction avec plusieurs paramètres référence ET retour référence | ✅ Oui |
| Struct contenant une référence | ✅ Oui |
| Trait object dans une struct | ✅ Oui (`Box<dyn Trait + 'a>`) |

---

## Résumé

Les lifetimes ne sont pas une punition — elles sont la formalisation de quelque chose que tout bon programmeur fait mentalement : "cette référence est-elle encore valide quand je l'utilise ?". Rust vous force à le rendre explicite là où ce n'est pas évident, et vérifie automatiquement là où ça l'est.

Avec la pratique, la plupart des codes n'ont besoin d'aucune annotation explicite. Et quand le compilateur en demande une, c'est souvent le signe que votre design mérite réflexion.
