# 0.1.2 — Ownership et Borrowing : Sécuriser vos références

> **Prérequis :** [0.1.1 — Stack vs Heap](./0.1.1-stack-vs-heap.md)
> **Durée estimée :** 60 minutes

## Le problème que Rust résout

Avant de comprendre l'ownership, comprenons le problème qu'il résout.

Dans les langages sans GC (C, C++), on peut écrire :

```c
// C — dangereux
char* create_string() {
    char buffer[100] = "bonjour";
    return buffer; // ← ERREUR : retourne un pointeur vers la stack locale
                   // Le buffer est détruit dès que la fonction retourne
}

// L'appelant a un dangling pointer — comportement indéfini
char* s = create_string();
printf("%s", s); // Peut afficher n'importe quoi, ou crasher
```

Ou pire :

```c
// C — double free
char* s = malloc(100);
free(s);
free(s); // ← Comportement indéfini, peut corrompre le tas
```

Ces erreurs sont responsables de la majorité des vulnérabilités de sécurité critiques dans les logiciels système. Rust les élimine entièrement — **à la compilation**, sans coût à l'exécution.

---

## Les trois règles de l'ownership

L'ownership en Rust repose sur trois règles simples. Le compilateur les vérifie pour chaque valeur :

> 1. **Chaque valeur a exactement un propriétaire.**
> 2. **Il ne peut y avoir qu'un seul propriétaire à la fois.**
> 3. **Quand le propriétaire quitte son scope, la valeur est libérée.**

Ces règles semblent contraignantes. En pratique, elles rendent le code plus prévisible et plus sûr.

---

## Le Move — transfert d'ownership

Quand vous assignez une valeur à une nouvelle variable, Rust **déplace** la valeur — il ne la copie pas.

```rust
fn main() {
    let s1 = String::from("bonjour");
    let s2 = s1; // s1 est DÉPLACÉ dans s2
    
    // println!("{}", s1); // ← ERREUR DE COMPILATION
    // error[E0382]: borrow of moved value: `s1`
    
    println!("{}", s2); // ✅ OK
}
```

Pourquoi ? Parce que permettre l'utilisation de `s1` après le move créerait deux propriétaires pour la même donnée heap — ce qui entraînerait un double free à la fin du scope.

### Visualisation du move

```
Avant le move :                    Après let s2 = s1 :

Stack           Heap               Stack           Heap
s1: ptr ──────> "bonjour"         s1: [invalide]
                                   s2: ptr ──────> "bonjour"
```

`s1` est marqué comme invalide. Le compilateur refuse toute utilisation ultérieure.

### Types qui se copient plutôt que de se déplacer

Les types primitifs implémentent le trait `Copy` — ils sont copiés, pas déplacés :

```rust
fn main() {
    let x = 5;
    let y = x; // COPIE — x est toujours valide
    
    println!("x={}, y={}", x, y); // ✅ Les deux sont utilisables
}
```

Règle pratique : si un type peut être copié en dupliquant ses octets sans effet de bord, il implémente `Copy`. Les scalaires (`i32`, `f64`, `bool`, `char`) et les tuples de types `Copy` sont `Copy`. Les types avec heap (`String`, `Vec`) ne le sont pas.

---

## Le Clone — copie explicite

Si vous voulez une vraie copie d'un type non-`Copy`, utilisez `.clone()` :

```rust
fn main() {
    let s1 = String::from("bonjour");
    let s2 = s1.clone(); // Copie profonde — alloue un nouveau buffer heap
    
    println!("s1={}, s2={}", s1, s2); // ✅ Les deux sont valides et indépendants
}
```

> ⚠️ **Piège courant :** utiliser `.clone()` partout pour éviter les erreurs du borrow checker. C'est une béquille — cherchez d'abord à comprendre pourquoi le compilateur refuse votre code.

---

## L'Ownership dans les fonctions

Passer une valeur à une fonction transfère l'ownership :

```rust
fn prend_ownership(s: String) {
    println!("{}", s);
} // ← s est droppé ici

fn main() {
    let s = String::from("bonjour");
    prend_ownership(s);
    // println!("{}", s); // ← ERREUR : s a été déplacé dans la fonction
}
```

C'est souvent trop restrictif. La solution : **le borrowing**.

---

## Le Borrowing — emprunter sans posséder

Un **emprunt** (borrow) est une référence à une valeur dont vous n'êtes pas propriétaire. La valeur originale reste valide chez son propriétaire.

### Emprunt immutable (&T)

```rust
fn affiche_longueur(s: &String) -> usize {
    s.len()
    // s n'est pas droppé ici — on n'en est pas propriétaire
}

fn main() {
    let s = String::from("bonjour");
    let longueur = affiche_longueur(&s); // On passe une référence
    
    println!("'{}' fait {} caractères", s, longueur); // ✅ s toujours valide
}
```

Règle : vous pouvez avoir **autant de références immutables que vous voulez** — simultanément.

```rust
fn main() {
    let s = String::from("bonjour");
    
    let r1 = &s;
    let r2 = &s;
    let r3 = &s;
    
    println!("{} {} {}", r1, r2, r3); // ✅ Parfaitement valide
}
```

### Emprunt mutable (&mut T)

Si vous avez besoin de modifier la valeur empruntée :

```rust
fn ajoute_monde(s: &mut String) {
    s.push_str(", monde");
}

fn main() {
    let mut s = String::from("bonjour");
    ajoute_monde(&mut s);
    println!("{}", s); // "bonjour, monde"
}
```

**Règle critique :** vous ne pouvez avoir qu'**une seule référence mutable** à la fois.

```rust
fn main() {
    let mut s = String::from("bonjour");
    
    let r1 = &mut s;
    // let r2 = &mut s; // ← ERREUR DE COMPILATION
    // error[E0499]: cannot borrow `s` as mutable more than once at a time
    
    println!("{}", r1);
}
```

---

## Les règles du Borrowing résumées

> 1. **Soit** plusieurs références immutables (`&T`) — autant que vous voulez.
> 2. **Soit** une seule référence mutable (`&mut T`).
> 3. **Jamais les deux en même temps.**

Ces règles empêchent les **data races** à la compilation. C'est la garantie la plus précieuse de Rust pour les systèmes concurrents.

## 🔥 Failure First — Provoquer une erreur de borrow

```rust
fn main() {
    let mut v = vec![1, 2, 3, 4, 5];
    
    // On prend une référence immutable
    let premier = &v[0];
    
    // On essaie de modifier le vecteur
    v.push(6); // ← ERREUR DE COMPILATION
    // error[E0502]: cannot borrow `v` as mutable because it is also borrowed as immutable
    
    println!("Le premier élément est {}", premier);
}
```

**Pourquoi le compilateur refuse ?**

`v.push(6)` peut entraîner une réallocation du buffer interne de `v`. Si cela se produit, le pointeur stocké dans `premier` devient un **dangling pointer** — il pointe vers de la mémoire libérée. Le compilateur l'empêche statiquement.

**Correction :**

```rust
fn main() {
    let mut v = vec![1, 2, 3, 4, 5];
    
    // Option 1 : copier la valeur avant de modifier
    let premier = v[0]; // i32 est Copy, donc pas de borrow
    v.push(6);
    println!("Le premier élément est {}", premier); // ✅
    
    // Option 2 : finir d'utiliser la référence avant de modifier
    {
        let premier_ref = &v[0];
        println!("Premier : {}", premier_ref);
    } // ← premier_ref n'existe plus
    v.push(7); // ✅ OK maintenant
}
```

## Le Non-Lexical Lifetime (NLL)

Depuis Rust 2018, le borrow checker comprend que la durée de vie d'un emprunt se termine à la **dernière utilisation**, pas à la fin du scope syntaxique :

```rust
fn main() {
    let mut s = String::from("bonjour");
    
    let r1 = &s;
    println!("{}", r1); // ← Dernière utilisation de r1
    // r1 n'existe plus "logiquement" après cette ligne
    
    let r2 = &mut s; // ✅ OK — r1 n'est plus actif
    r2.push_str(", monde");
    println!("{}", r2);
}
```

C'est une amélioration majeure : le code naturel compile souvent sans avoir à ajouter de blocs `{}` artificiels.

## Ownership dans les structs

```rust
struct Utilisateur {
    nom: String,       // Possède la String
    email: String,     // Possède la String
    actif: bool,       // Copy — pas de heap
}

fn main() {
    let user1 = Utilisateur {
        nom: String::from("Alice"),
        email: String::from("alice@example.com"),
        actif: true,
    };
    
    // Update syntax — déplace les champs String non copiés
    let user2 = Utilisateur {
        email: String::from("bob@example.com"),
        ..user1 // ← nom est DÉPLACÉ depuis user1
    };
    
    // println!("{}", user1.nom); // ← ERREUR : nom a été déplacé
    println!("{}", user2.nom);    // ✅
    println!("{}", user2.email);  // ✅
}
```

## 🔬 Sous le capot — Comment le compilateur vérifie l'ownership

Le compilateur Rust construit un **graphe de durées de vie** pour chaque valeur. Il analyse statiquement tous les chemins d'exécution possibles et vérifie :

1. Qu'une valeur déplacée n'est plus utilisée.
2. Qu'une référence ne "survit" pas à la valeur qu'elle référence.
3. Que les règles mutable/immutable ne sont pas violées.

Ce travail se fait entièrement à la compilation. À l'exécution, il n'y a aucune vérification — c'est pourquoi Rust est aussi rapide que C.

## 🛠️ Patterns courants en production

### Pattern : retourner une référence vers les données d'une structure

```rust
struct Cache {
    data: HashMap<String, String>,
}

impl Cache {
    fn get(&self, key: &str) -> Option<&String> {
        // On retourne une référence vers les données du Cache
        // La durée de vie de la référence est liée à &self
        self.data.get(key)
    }
}

fn main() {
    let mut cache = Cache {
        data: HashMap::new(),
    };
    cache.data.insert("clé".to_string(), "valeur".to_string());
    
    if let Some(val) = cache.get("clé") {
        println!("Trouvé : {}", val);
        // val vit aussi longtemps que cache est accessible
    }
}
```

### Pattern : éviter les clones inutiles avec des slices

```rust
// ❌ Inefficace : prend ownership et ne le rend pas
fn longueur_propriete(s: String) -> usize {
    s.len()
}

// ✅ Efficace : emprunte seulement ce dont on a besoin
fn longueur_emprunt(s: &str) -> usize {
    s.len()
}

fn main() {
    let s = String::from("bonjour");
    
    // &str est plus flexible : fonctionne avec String ET les string literals
    println!("{}", longueur_emprunt(&s));      // String → &str
    println!("{}", longueur_emprunt("monde")); // &str → &str
}
```

## Résumé

| Concept | Règle | Raison |
|---|---|---|
| Move | Une valeur a un seul propriétaire | Évite le double free |
| Copy | Types simples sont copiés | Pas de gestion mémoire nécessaire |
| &T (borrow immutable) | Plusieurs simultanément OK | Lecture seule = pas de race |
| &mut T (borrow mutable) | Un seul à la fois | Évite les data races |
| Drop | Automatique à la fin du scope | RAII — ressources toujours libérées |
