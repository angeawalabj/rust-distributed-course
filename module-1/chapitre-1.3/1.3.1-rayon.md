## Concurrence vs Parallélisme — La distinction fondamentale

Jusqu'ici, nous avons étudié la **concurrence** avec Tokio : gérer de nombreuses tâches qui progressent entrelacées sur un nombre limité de threads. C'est idéal pour l'I/O.

Le **parallélisme** est différent : exécuter plusieurs calculs **simultanément** sur plusieurs cœurs CPU. C'est ce que Rayon apporte.

```
Concurrence (Tokio) :
Thread 1 :  [Tâche A] -- attente I/O -- [Tâche B] -- [Tâche A continue]
Utilisation CPU : ~20% par thread (la plupart du temps en attente)

Parallélisme (Rayon) :
Thread 1 :  [Calcul 1/4 du tableau ─────────────────────────────────]
Thread 2 :  [Calcul 2/4 du tableau ─────────────────────────────────]
Thread 3 :  [Calcul 3/4 du tableau ─────────────────────────────────]
Thread 4 :  [Calcul 4/4 du tableau ─────────────────────────────────]
Utilisation CPU : ~100% sur tous les cœurs
```

## Rayon — Work-stealing parallelism

```toml
[dependencies]
rayon = "1"
```

L'API de Rayon est conçue pour être une substitution quasi-transparente des itérateurs standard :

```rust
use rayon::prelude::*;

fn main() {
    let données: Vec<u64> = (1..=10_000_000).collect();
    
    // Version séquentielle
    let somme_seq: u64 = données.iter().sum();
    
    // Version parallèle — une seule différence : .par_iter() au lieu de .iter()
    let somme_par: u64 = données.par_iter().sum();
    
    assert_eq!(somme_seq, somme_par);
    println!("Somme : {}", somme_par);
}
```

Rayon divise automatiquement les données entre les threads disponibles, utilise le **work-stealing** pour équilibrer la charge, et rassemble les résultats.

## Opérations parallèles courantes

```rust
use rayon::prelude::*;

fn traitement_complet(données: Vec<i64>) {
    // map parallèle
    let carrés: Vec<i64> = données.par_iter()
        .map(|&x| x * x)
        .collect();
    
    // filter + collect parallèles
    let pairs: Vec<i64> = données.par_iter()
        .filter(|&&x| x % 2 == 0)
        .copied()
        .collect();
    
    // fold parallèle (reduce avec état local par thread)
    let somme: i64 = données.par_iter()
        .fold(|| 0i64, |acc, &x| acc + x)
        .sum(); // Combinaison finale des résultats partiels
    
    // Avec opérations enchaînées
    let résultat: Vec<String> = données
        .par_iter()
        .filter(|&&x| x > 0)
        .map(|&x| x * 2)
        .filter(|&x| x < 1000)
        .map(|x| format!("item_{}", x))
        .collect();
    
    // Tri parallèle
    let mut à_trier = données.clone();
    à_trier.par_sort();
    à_trier.par_sort_unstable(); // Plus rapide si l'ordre d'égaux importe peu
    
    println!("Paires: {}, Somme: {}", pairs.len(), somme);
}
```

## Traitement de fichiers en parallèle

```rust
use rayon::prelude::*;
use std::path::{Path, PathBuf};
use std::fs;

fn analyser_logs(répertoire: &Path) -> Vec<ErreurLog> {
    // Lister tous les fichiers
    let fichiers: Vec<PathBuf> = fs::read_dir(répertoire)
        .unwrap()
        .filter_map(|e| e.ok())
        .map(|e| e.path())
        .filter(|p| p.extension().map_or(false, |e| e == "log"))
        .collect();
    
    // Analyser tous les fichiers en parallèle
    fichiers
        .par_iter()
        .flat_map(|chemin| analyser_fichier_log(chemin))
        .collect()
}

fn analyser_fichier_log(chemin: &Path) -> Vec<ErreurLog> {
    let contenu = fs::read_to_string(chemin).unwrap_or_default();
    
    contenu
        .lines()
        .filter(|ligne| ligne.contains("ERROR") || ligne.contains("FATAL"))
        .map(|ligne| ErreurLog::parser(ligne))
        .collect()
}

#[derive(Debug)]
struct ErreurLog {
    niveau: String,
    message: String,
}

impl ErreurLog {
    fn parser(ligne: &str) -> Self {
        // Parsing simplifié
        ErreurLog {
            niveau: if ligne.contains("FATAL") { "FATAL".into() } else { "ERROR".into() },
            message: ligne.to_string(),
        }
    }
}
```

## Benchmarking : séquentiel vs parallèle

```rust
// benches/rayon_benchmark.rs
use criterion::{black_box, criterion_group, criterion_main, Criterion};
use rayon::prelude::*;

fn traitement_intensif(n: u64) -> u64 {
    // Simulation d'un calcul coûteux
    (0..n).fold(0u64, |acc, x| acc.wrapping_add(x.wrapping_mul(x)))
}

fn benchmark_séquentiel(c: &mut Criterion) {
    let données: Vec<u64> = (0..10_000).collect();
    
    c.bench_function("séquentiel", |b| b.iter(|| {
        données.iter()
            .map(|&x| traitement_intensif(black_box(x)))
            .sum::<u64>()
    }));
}

fn benchmark_parallèle(c: &mut Criterion) {
    let données: Vec<u64> = (0..10_000).collect();
    
    c.bench_function("parallèle rayon", |b| b.iter(|| {
        données.par_iter()
            .map(|&x| traitement_intensif(black_box(x)))
            .sum::<u64>()
    }));
}

criterion_group!(benches, benchmark_séquentiel, benchmark_parallèle);
criterion_main!(benches);
```

Sur une machine 8 cœurs, résultat typique :
```
séquentiel          time: [48.2 ms 48.5 ms 48.8 ms]
parallèle rayon     time: [6.8 ms  6.9 ms  7.0 ms]
# Accélération : ~7x sur 8 cœurs (overhead Rayon ≈ 1 cœur)
```

## Rayon + Tokio — Combiner I/O et CPU

Dans une application réelle, vous aurez souvent besoin des deux :

```rust
use rayon::prelude::*;
use tokio::task;

// Traitement hybride : I/O async + CPU parallèle
async fn pipeline_complet(urls: Vec<String>) -> Vec<String> {
    // Phase 1 : I/O async — télécharger tous les fichiers en parallèle
    let mut téléchargements = tokio::task::JoinSet::new();
    for url in urls {
        téléchargements.spawn(async move {
            reqwest::get(&url).await?.bytes().await
        });
    }
    
    let mut données_brutes = vec![];
    while let Some(résultat) = téléchargements.join_next().await {
        if let Ok(Ok(bytes)) = résultat {
            données_brutes.push(bytes.to_vec());
        }
    }
    
    // Phase 2 : traitement CPU — analyser en parallèle avec Rayon
    // spawn_blocking pour ne pas bloquer le runtime Tokio
    task::spawn_blocking(move || {
        données_brutes
            .par_iter()
            .map(|données| traiter_données_cpu(données))
            .collect()
    }).await.unwrap()
}

fn traiter_données_cpu(données: &[u8]) -> String {
    // Traitement CPU-intensif : parsing, compression, chiffrement...
    format!("traité: {} octets", données.len())
}
```

## 🔥 Failure First — Quand Rayon est contre-productif

```rust
use rayon::prelude::*;
use std::time::Instant;

fn main() {
    // ❌ Rayon sur des petits datasets — l'overhead dépasse le gain
    let petites_données: Vec<i32> = (0..100).collect();
    
    let début = Instant::now();
    let _: i32 = petites_données.par_iter().sum();
    println!("Parallèle (100 éléments) : {:?}", début.elapsed());
    // ~50 µs (overhead de création des threads)
    
    let début = Instant::now();
    let _: i32 = petites_données.iter().sum();
    println!("Séquentiel (100 éléments) : {:?}", début.elapsed());
    // ~200 ns (150x plus rapide !)
    
    // ✅ Rayon est rentable à partir de ~10 000 éléments
    // pour des opérations légères, ou moins pour des calculs lourds
}
```

**Règle pratique :** Benchmarkez avant d'utiliser Rayon. L'overhead de création des threads (~10-50 µs) doit être négligeable par rapport au travail effectué.

## Configuration du pool de threads Rayon

```rust
use rayon::ThreadPoolBuilder;

fn main() {
    // Construire un pool dédié avec configuration personnalisée
    let pool = ThreadPoolBuilder::new()
        .num_threads(4)             // Utiliser seulement 4 cœurs
        .thread_name(|i| format!("rayon-worker-{}", i))
        .stack_size(4 * 1024 * 1024) // 4 Mo de stack par thread
        .build()
        .unwrap();
    
    let résultat: Vec<i32> = pool.install(|| {
        (0..1_000_000i32).into_par_iter().map(|x| x * 2).collect()
    });
    
    println!("{} éléments traités", résultat.len());
}
```

## Résumé

| Situation | Outil |
|---|---|
| I/O concurrent (réseau, disque) | Tokio async |
| CPU-intensif sur grande collection | Rayon |
| I/O async + traitement CPU | Tokio + `spawn_blocking` + Rayon |
| Petite collection (<1000 éléments) | Itérateurs standard |
| Tri de grande collection | `par_sort()` |
