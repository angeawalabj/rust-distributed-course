# 1.1.1 — Futures, Poll et Waker : Comprendre l'async de l'intérieur

> **Prérequis :** [Module 0 complet](../../module-0/)
> **Durée estimée :** 60 minutes

## Pourquoi l'asynchronisme est indispensable pour les serveurs modernes

Imaginez un serveur HTTP classique avec threading synchrone : chaque connexion cliente reçoit un thread dédié. Pour 10 000 connexions simultanées, vous avez 10 000 threads — chacun consommant environ 2 Mo de stack. C'est **20 Go de RAM** juste pour les stacks.

Et la plupart du temps, ces threads ne font rien — ils attendent une réponse de la base de données, une lecture réseau, une écriture disque. Ce modèle est massivement inefficace.

L'asynchronisme résout ce problème : un seul thread peut gérer des milliers de connexions en attendant les I/O, passant d'une tâche à l'autre sans bloquer.

---

## Threads OS vs Tâches asynchrones

```
Modèle threading (1 thread par connexion) :
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│Thread 1│ │Thread 2│ │Thread 3│ │ ...    │
│ 2 Mo   │ │ 2 Mo   │ │ 2 Mo   │ │        │
│[attend]│ │[attend]│ │[attend]│ │        │
└────────┘ └────────┘ └────────┘ └────────┘
→ 10 000 connexions = 20 Go RAM, OS scheduler surchargé

Modèle async (N tâches sur M threads) :
┌─────────────────────────────────┐
│         Thread 1 (tokio)        │
│  Tâche1 → Tâche2 → Tâche3 → .. │
│  [I/O? → suspension → reprise]  │
└─────────────────────────────────┘
→ 10 000 connexions = quelques Mo, une poignée de threads
```

## Les Futures — La brique de base de l'async

En Rust, une **Future** est un calcul qui peut ne pas être terminé immédiatement. C'est une valeur qui représente "quelque chose qui va éventuellement produire un résultat".

```rust
// Le trait Future, simplifié
pub trait Future {
    type Output;
    
    // Appelé par le runtime pour progresser la future
    // Retourne Poll::Ready(val) si terminé
    // Retourne Poll::Pending si encore en attente
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}
```

> 💡 **Important :** Une future en Rust ne fait rien par elle-même. Elle doit être **exécutée** par un runtime (comme Tokio). C'est une description de calcul, pas le calcul lui-même.

## async/await — La syntaxe haut niveau

Écrire des futures à la main est laborieux. `async`/`await` est du sucre syntaxique qui génère automatiquement le code de la machine d'état :

```rust
// Avec async/await — lisible
async fn lire_fichier(chemin: &str) -> Result<String, std::io::Error> {
    let contenu = tokio::fs::read_to_string(chemin).await?;
    Ok(contenu)
}

// Ce que le compilateur génère (conceptuellement) :
// Une struct qui implémente Future et contient l'état de progression
// enum LireFichierFuture {
//     DépartLecture { chemin: String },
//     AttenteIO { handle: tokio::fs::ReadToString },
//     Terminé,
// }
```

### Règles d'utilisation

```rust
// `async fn` transforme la fonction pour qu'elle retourne une Future
async fn ma_fonction() -> i32 {
    42
}

// `await` suspend la fonction jusqu'à ce que la future soit prête
async fn appel() -> i32 {
    let val = ma_fonction().await; // Suspension possible ici
    val + 1
}

// async peut être utilisé dans des blocs
async fn avec_bloc() -> i32 {
    let résultat = async {
        // Ce bloc est lui-même une future
        10 + 20
    }.await;
    résultat
}
```

## Tokio — Le runtime asynchrone

`async`/`await` ne s'exécute pas tout seul. Il faut un **runtime** qui pilote les futures. Tokio est le runtime le plus utilisé en Rust.

```toml
# Cargo.toml
[dependencies]
tokio = { version = "1", features = ["full"] }
```

```rust
// Le point d'entrée async avec Tokio
#[tokio::main]
async fn main() {
    println!("Bonjour depuis l'async !");
    
    // Lancer une tâche concurrente
    let handle = tokio::spawn(async {
        // Cette tâche s'exécute concurremment
        tokio::time::sleep(std::time::Duration::from_millis(100)).await;
        42
    });
    
    // Faire autre chose pendant que la tâche tourne...
    println!("Pendant ce temps...");
    
    // Attendre le résultat
    let résultat = handle.await.unwrap();
    println!("Résultat : {}", résultat);
}
```

### Macro `#[tokio::main]` — Ce qu'elle fait

```rust
// Ce que vous écrivez
#[tokio::main]
async fn main() { /* ... */ }

// Ce que le compilateur génère (simplifié)
fn main() {
    tokio::runtime::Builder::new_multi_thread()
        .enable_all()
        .build()
        .unwrap()
        .block_on(async {
            /* votre code async ici */
        });
}
```

## La concurrence avec join! et select!

### `join!` — Attendre plusieurs futures en parallèle

```rust
use tokio::time::{sleep, Duration};

async fn tâche_rapide() -> &'static str {
    sleep(Duration::from_millis(100)).await;
    "rapide"
}

async fn tâche_lente() -> &'static str {
    sleep(Duration::from_millis(500)).await;
    "lente"
}

#[tokio::main]
async fn main() {
    let début = std::time::Instant::now();
    
    // Les deux tâches s'exécutent CONCURREMMENT
    let (r1, r2) = tokio::join!(tâche_rapide(), tâche_lente());
    
    println!("{} et {} en {:?}", r1, r2, début.elapsed());
    // Affiche "rapide et lente en ~500ms" — pas 600ms
}
```

### `select!` — Attendre la première future prête

```rust
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    tokio::select! {
        _ = sleep(Duration::from_millis(100)) => {
            println!("Timeout ! La requête a pris trop longtemps.");
        }
        résultat = opération_longue() => {
            println!("Résultat reçu : {}", résultat);
        }
    }
}

async fn opération_longue() -> String {
    sleep(Duration::from_millis(500)).await;
    "résultat".to_string()
}
```

## 🔥 Failure First — Bloquer le runtime async

L'erreur la plus courante : appeler du code bloquant dans un contexte async.

```rust
use std::time::Duration;

#[tokio::main]
async fn main() {
    println!("Avant le blocage");
    
    // ❌ MAUVAIS : std::thread::sleep bloque le thread Tokio
    // Pendant ce temps, AUCUNE autre tâche ne peut s'exécuter sur ce thread
    std::thread::sleep(Duration::from_secs(5)); // Bloque tout !
    
    println!("Après le blocage");
}
```

**Symptôme en production :** Votre serveur répond de plus en plus lentement, puis ne répond plus — sans que les logs montrent d'erreur évidente. C'est le "thread starvation" : le runtime est bloqué à traiter une tâche lente.

```rust
// ✅ Solution 1 : utiliser tokio::time::sleep (non-bloquant)
tokio::time::sleep(Duration::from_secs(5)).await;

// ✅ Solution 2 : déporter le code bloquant hors du runtime
tokio::task::spawn_blocking(|| {
    // Ce code s'exécute dans un pool de threads dédié
    std::thread::sleep(Duration::from_secs(5));
    "résultat du code bloquant"
}).await.unwrap();

// ✅ Solution 3 : utiliser les APIs async (tokio::fs plutôt que std::fs)
let contenu = tokio::fs::read_to_string("fichier.txt").await?;
// Plutôt que :
// let contenu = std::fs::read_to_string("fichier.txt")?; // BLOQUANT
```

## 🔬 Sous le capot — Le mécanisme de Poll

Voici ce que le runtime fait réellement :

```
1. Runtime appelle future.poll(cx)
2. La future tente de progresser :
   ├─ Résultat disponible immédiatement → Poll::Ready(valeur)
   └─ En attente de I/O → 
      a. Enregistre un Waker auprès du OS (epoll/kqueue/IOCP)
      b. Retourne Poll::Pending
3. Le runtime passe à une autre future prête
4. L'OS notifie via le Waker quand l'I/O est prête
5. Le runtime re-poll cette future
6. La future peut maintenant avancer → Poll::Ready
```

C'est ce mécanisme qui permet à un seul thread de gérer des milliers de connexions sans jamais "attendre" activement.

## Cancellation — Annuler une future

Les futures Rust sont annulables — si vous droppez une future, elle s'arrête proprement :

```rust
use tokio::time::{sleep, timeout, Duration};

#[tokio::main]
async fn main() {
    // timeout wraps une future avec une durée maximale
    let résultat = timeout(
        Duration::from_millis(100),
        opération_très_longue()
    ).await;
    
    match résultat {
        Ok(val) => println!("Terminé : {}", val),
        Err(_) => println!("Timeout — opération annulée"),
    }
}

async fn opération_très_longue() -> i32 {
    sleep(Duration::from_secs(10)).await;
    42
}
```

## Résumé

| Concept | Rôle |
|---|---|
| `Future` | Description d'un calcul asynchrone |
| `async fn` | Sucre syntaxique — génère une Future |
| `.await` | Suspend jusqu'à ce que la Future soit prête |
| Runtime (Tokio) | Pilote l'exécution des Futures |
| `tokio::spawn` | Lance une tâche concurrente |
| `join!` | Attend plusieurs futures en parallèle |
| `select!` | Attend la première future prête |
| `spawn_blocking` | Déporte le code bloquant hors du runtime |

**Règle d'or :** Dans un contexte async, n'appelez jamais de code bloquant sans `spawn_blocking`. Utilisez toujours les équivalents async (tokio::fs, tokio::net, tokio::time).
