## Les trois piliers de l'observabilité

Un système observable est un système que vous pouvez comprendre de l'extérieur, sans modifier le code. Les trois piliers sont :

```
Logs    → "Que s'est-il passé ?"         (événements discrets)
Métriques → "Dans quel état est le système ?" (agrégats numériques)
Traces  → "Où est passée cette requête ?" (chemin d'exécution)

Sans observabilité :
  Incident → "Je ne sais pas pourquoi"  → heures de débogage
Avec observabilité :
  Incident → dashboard → 30 secondes pour identifier la cause
```

## Pilier 1 : Logs structurés avec `tracing`

Les logs en texte libre (`println!("erreur: {}", e)`) sont inutilisables en production. Les logs structurés (champs clé/valeur, format JSON) sont indexables, filtrables et exploitables.

```toml
[dependencies]
tracing = "0.1"
tracing-subscriber = { version = "0.3", features = ["env-filter", "json"] }
tracing-bunyan-formatter = "0.3"
```

```rust
// src/télémétrie.rs
use tracing_subscriber::{layer::SubscriberExt, util::SubscriberInitExt, EnvFilter};

pub fn initialiser_tracing(service: &str, env: &str) {
    let filtre = EnvFilter::try_from_default_env()
        .unwrap_or_else(|_| "info".into());

    let registre = tracing_subscriber::registry().with(filtre);

    if env == "production" {
        // JSON structuré pour les agrégateurs (Datadog, Elastic, Loki)
        registre
            .with(
                tracing_bunyan_formatter::BunyanFormattingLayer::new(
                    service.to_string(),
                    std::io::stdout,
                )
            )
            .init();
    } else {
        // Format lisible pour le développement
        registre
            .with(tracing_subscriber::fmt::layer().pretty())
            .init();
    }
}
```

### Spans et événements

```rust
use tracing::{info, warn, error, debug, instrument, Span};

// @instrument : crée automatiquement un span autour de la fonction
#[instrument(
    name = "traiter_paiement",
    skip(config, connexion_db),  // Ne pas loguer les secrets
    fields(
        montant = %montant,
        devise = %devise,
        utilisateur_id = tracing::field::Empty, // Rempli plus tard
    )
)]
async fn traiter_paiement(
    montant: f64,
    devise: &str,
    config: &Config,
    connexion_db: &DbPool,
) -> Result<String, ErreurPaiement> {

    // Enrichir le span courant avec des données calculées
    Span::current().record("utilisateur_id", &"usr_42");

    debug!(étape = "validation", "Début du traitement");

    if montant <= 0.0 {
        warn!(montant = montant, "Montant négatif ou nul refusé");
        return Err(ErreurPaiement::MontantInvalide);
    }

    let id_transaction = uuid::Uuid::new_v4().to_string();
    info!(
        transaction_id = %id_transaction,
        montant = montant,
        devise = devise,
        "Paiement accepté"
    );

    Ok(id_transaction)
}
```

## Pilier 2 : Métriques Prometheus

Prometheus utilise un modèle de **scraping** : votre service expose un endpoint `/metrics` que Prometheus interroge périodiquement.

```toml
[dependencies]
prometheus = { version = "0.13", features = ["process"] }
axum = "0.7"
```

```rust
// src/métriques.rs
use prometheus::{
    Counter, CounterVec, Gauge, Histogram, HistogramOpts, HistogramVec,
    IntCounter, IntCounterVec, IntGauge, Opts, Registry,
};
use std::sync::Arc;

#[derive(Clone)]
pub struct Métriques {
    pub requêtes_totales: IntCounterVec,
    pub requêtes_actives: IntGauge,
    pub durée_requête: HistogramVec,
    pub erreurs_totales: IntCounterVec,
    pub connexions_db: IntGauge,
    pub taille_cache: IntGauge,
}

impl Métriques {
    pub fn nouvelles(registre: &Registry) -> Self {
        let requêtes_totales = IntCounterVec::new(
            Opts::new("http_requêtes_total", "Nombre total de requêtes HTTP")
                .namespace("mon_service"),
            &["méthode", "chemin", "statut"],
        ).unwrap();

        let requêtes_actives = IntGauge::with_opts(
            Opts::new("http_requêtes_actives", "Requêtes HTTP en cours")
                .namespace("mon_service"),
        ).unwrap();

        let durée_requête = HistogramVec::new(
            HistogramOpts::new("http_durée_secondes", "Durée des requêtes HTTP")
                .namespace("mon_service")
                .buckets(vec![0.001, 0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5]),
            &["méthode", "chemin"],
        ).unwrap();

        let erreurs_totales = IntCounterVec::new(
            Opts::new("erreurs_total", "Nombre total d'erreurs")
                .namespace("mon_service"),
            &["type", "code"],
        ).unwrap();

        let connexions_db = IntGauge::with_opts(
            Opts::new("db_connexions_actives", "Connexions DB actives")
                .namespace("mon_service"),
        ).unwrap();

        let taille_cache = IntGauge::with_opts(
            Opts::new("cache_entrées", "Nombre d'entrées en cache")
                .namespace("mon_service"),
        ).unwrap();

        registre.register(Box::new(requêtes_totales.clone())).unwrap();
        registre.register(Box::new(requêtes_actives.clone())).unwrap();
        registre.register(Box::new(durée_requête.clone())).unwrap();
        registre.register(Box::new(erreurs_totales.clone())).unwrap();
        registre.register(Box::new(connexions_db.clone())).unwrap();
        registre.register(Box::new(taille_cache.clone())).unwrap();

        Métriques {
            requêtes_totales,
            requêtes_actives,
            durée_requête,
            erreurs_totales,
            connexions_db,
            taille_cache,
        }
    }
}

// Middleware Axum qui instrumente chaque requête
pub async fn middleware_métriques(
    axum::extract::State(métriques): axum::extract::State<Arc<Métriques>>,
    req: axum::extract::Request,
    suivant: axum::middleware::Next,
) -> axum::response::Response {
    let méthode = req.method().to_string();
    let chemin = req.uri().path().to_string();
    let début = std::time::Instant::now();

    métriques.requêtes_actives.inc();

    let réponse = suivant.run(req).await;

    métriques.requêtes_actives.dec();

    let durée = début.elapsed().as_secs_f64();
    let statut = réponse.status().as_u16().to_string();

    métriques.requêtes_totales
        .with_label_values(&[&méthode, &chemin, &statut])
        .inc();

    métriques.durée_requête
        .with_label_values(&[&méthode, &chemin])
        .observe(durée);

    réponse
}

// Endpoint /metrics pour Prometheus
pub async fn handler_métriques(
    axum::extract::State(registre): axum::extract::State<Arc<Registry>>,
) -> axum::response::Response {
    use axum::response::IntoResponse;
    use prometheus::Encoder;

    let encoder = prometheus::TextEncoder::new();
    let familles = registre.gather();
    let mut buffer = Vec::new();
    encoder.encode(&familles, &mut buffer).unwrap();

    (
        [("content-type", "text/plain; version=0.0.4")],
        buffer,
    ).into_response()
}
```

        expr: |
          histogram_quantile(0.99,

          ) > 1.0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "P99 latence > 1 seconde"

## Pilier 3 : Tracing distribué avec OpenTelemetry

Le tracing distribué permet de suivre une requête à travers plusieurs microservices.

```toml
[dependencies]
opentelemetry = { version = "0.22", features = ["trace"] }
opentelemetry-otlp = { version = "0.15", features = ["grpc-tonic"] }
opentelemetry_sdk = { version = "0.22", features = ["rt-tokio"] }
tracing-opentelemetry = "0.23"
```

```rust
// src/télémétrie.rs (version complète avec OTLP)
use opentelemetry::global;
use opentelemetry_otlp::WithExportConfig;
use opentelemetry_sdk::{runtime, trace as sdktrace};
use tracing_subscriber::{layer::SubscriberExt, util::SubscriberInitExt};

pub fn initialiser_avec_otlp(service: &str, endpoint_otlp: &str) -> anyhow::Result<()> {
    // Configurer l'exportateur OTLP (vers Jaeger, Grafana Tempo, etc.)
    let traceur = opentelemetry_otlp::new_pipeline()
        .tracing()
        .with_exporter(
            opentelemetry_otlp::new_exporter()
                .tonic()
                .with_endpoint(endpoint_otlp)
        )
        .with_trace_config(
            sdktrace::config()
                .with_resource(opentelemetry_sdk::Resource::new(vec![
                    opentelemetry::KeyValue::new("service.name", service.to_string()),
                    opentelemetry::KeyValue::new("service.version", env!("CARGO_PKG_VERSION")),
                ]))
                .with_sampler(sdktrace::Sampler::AlwaysOn)
        )
        .install_batch(runtime::Tokio)?;

    // Connecter tracing → OpenTelemetry
    let couche_otel = tracing_opentelemetry::layer().with_tracer(traceur);

    tracing_subscriber::registry()
        .with(tracing_subscriber::EnvFilter::from_default_env())
        .with(tracing_subscriber::fmt::layer().json())
        .with(couche_otel)
        .init();

    Ok(())
}

// Propager le contexte de trace entre services
pub fn extraire_contexte_http(en_têtes: &axum::http::HeaderMap) -> opentelemetry::Context {
    use opentelemetry::propagation::Extractor;

    struct ExtracteurEnTêtes<'a>(&'a axum::http::HeaderMap);
    impl<'a> Extractor for ExtracteurEnTêtes<'a> {
        fn get(&self, key: &str) -> Option<&str> {
            self.0.get(key).and_then(|v| v.to_str().ok())
        }
        fn keys(&self) -> Vec<&str> {
            self.0.keys().map(|k| k.as_str()).collect()
        }
    }

    global::get_text_map_propagator(|p| p.extract(&ExtracteurEnTêtes(en_têtes)))
}
```

## 🔥 Failure First — Identifier une fuite mémoire via les métriques

```rust
// Bug intentionnel : cache qui grossit sans limite
lazy_static::lazy_static! {
    static ref CACHE_BRISÉ: std::sync::Mutex<Vec<String>> =
        std::sync::Mutex::new(Vec::new());
}

async fn handler_avec_fuite(
    métriques: axum::extract::State<Arc<Métriques>>,
    req: axum::extract::Request,
) -> &'static str {
    // Fuite : on ajoute sans jamais nettoyer
    CACHE_BRISÉ.lock().unwrap().push("entrée".repeat(1000));
    métriques.taille_cache.set(CACHE_BRISÉ.lock().unwrap().len() as i64);
    "OK"
}
```

Dans Grafana, vous verrez `mon_service_cache_entrées` croître linéairement sans jamais redescendre — signal classique d'une fuite mémoire.
